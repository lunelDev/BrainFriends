# Source GMP Requirements Matrix

작성일: 2026-05-08  
원문: `디지털의료기기+GMP+가이드라인(민원인+안내서)(최종).pdf`  
목적: 디지털의료기기 GMP 원문에서 브레인프렌즈 개발·품질·출시 준비에 직접 걸리는 요구사항을 추출해 작업 단위로 고정한다.

## 1. 해석 원칙

- 이 문서는 기능 명세가 아니라 품질시스템 증빙 요구사항이다.
- 브레인프렌즈가 SaMD로 정리되는 경우 `별표2 + 별표3` 축을 기본으로 보고, AI/ML 기능을 성능 클레임으로 유지하면 `별표4`까지 준비한다.
- 원문은 제조업 허가, GMP 적합인정, 품목 인허가를 분리해서 보게 만든다. 코드가 있어도 QMS 기록이 없으면 GMP 증빙으로 부족하다.
- NIDS 상담노트는 실행 순서를 정하는 내부 참고이고, 실제 제출 근거는 이 GMP 가이드라인과 관련 법령/고시가 된다.

## 2. 브레인프렌즈 적용 결론

브레인프렌즈의 다음 개발 우선순위는 기능 추가보다 아래 6개 품질 증빙을 코드와 연결하는 것이다.

1. 요구사항-설계-구현-시험-위험통제 추적성
2. SOUP/SBOM/라이선스/CVE 관리
3. 변경요청-승인-구현-검증-릴리스 기록
4. 문제보고-CAPA-시판 후 피드백 처리
5. 출시 전 잔여위험, 검증완료, 버전, 패키징 기록
6. AI/STT/안면/시선/AAC 모델·알고리즘 사양과 성능평가 기록

## 6. PM 결론

GMP 관점에서 브레인프렌즈의 약한 부분은 기능이 아니라 품질 기록이다. 지금부터는 개발 완료의 정의를 “화면이 동작한다”가 아니라 “요구사항 ID, 위험 ID, 테스트 결과, 릴리스 버전, 문제보고 기록으로 추적된다”로 바꿔야 한다.

다음 개발 작업은 `release manifest` 확장과 `문제보고/CAPA export`가 우선이다. 이 두 개가 잡히면 현재 앱에서 발생하는 저장 실패, 채점 이상, STT 실패, 권한 거부를 GMP 증빙 흐름으로 연결할 수 있다.
