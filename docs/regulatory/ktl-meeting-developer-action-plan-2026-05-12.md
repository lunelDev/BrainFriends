개발팀이 바로 반영해야 할 방향은 다음 6개다.

| 우선순위 | 개발 항목 | 이유 | 현재 상태 |
| --- | --- | --- | --- |
| P0 | 1등급/2등급 표현 분기 | 1등급 신고 트랙은 치료·재활·진단 표현을 피해야 함 | 일부 문구 정리됨. 전체 UI/문서 재점검 필요 |
| P0 | V 모델 문서 세트화 | KTL이 연결 추적성을 가장 중요하게 봄 | SR/TC/traceability 일부 있음. 문서 세트 분리 필요 |

## 2. 제품 전략이 개발에 주는 영향

KTL 권고는 **1등급 우선 출시 + 2등급 장기 준비**다.

개발적으로는 제품을 하나로만 보면 안 된다. 같은 코드베이스 안에서 다음 두 모드를 분리해 관리해야 한다.

| 구분 | 1등급 신고용 표현 | 2등급 준비용 표현 |
| --- | --- | --- |
| 목적 | 정보 제공, 관리, 훈련 보조, 발화 보조 | 치료, 재활, 증상 개선 |
| 결과 | 참고 지표, 수행 기록, 치료사 검토 보조 | 임상 성능 지표, 치료 효과 검증 |
| AI 역할 | 자동 판단 아님, 보조 지표 | 성능평가 대상 AI 구성요소 |
| UI 문구 | 자가점검, 훈련 보조, 의사표현 보조 | 언어재활, 치료 효과, 개선 |
| 개발 필요 | claim-lock 1등급 모드 추가 | 기존 2등급 SaMD 문서 유지 |

### 개발 작업

- `claim-lock.md`에 **1등급 신고용 금지어/허용어 표** 추가
- 사용자 화면 문구를 `regulatoryMode = class1_info | class2_samd` 기준으로 분기할 수 있게 준비
- 리포트 결과 문구는 기본적으로 “진단/치료 결정 아님, 치료사 검토용 보조 지표”로 고정
- “재활”, “치료”, “효과”, “개선”, “진단” 표현은 1등급 모드에서 노출되지 않게 점검

## 3. V 모델 문서화 작업

KTL이 가장 강조한 것은 **연결 추적성**이다.

요구사항이 어디 설계에 반영됐고, 어떤 코드/모듈로 구현됐으며, 어떤 테스트로 검증됐는지 한 줄로 이어져야 한다.

### 문서 세트

KTL 권고 기준으로 하나의 거대한 문서보다 최소 8~10개 문서로 나눠야 한다.

| 문서 | 개발팀 산출물 | 현재 근거 | 부족한 점 |
| --- | --- | --- | --- |
| SW 개발 계획서 | 범위, 일정, 역할, 형상관리, 검증 독립성 | 일부 문서 존재 | 검증자 독립성/적격성 기준 필요 |
| 사용자 요구사항 | 환자/치료사/관리자/보호자 요구사항 | 일부 기획 문서 | 1등급/2등급 목적별 분리 필요 |
| SW 요구사항 명세서 | SR-* 요구사항 | `src/lib/vnv/requirements.ts`, `docs/regulatory/srs.md` | KTL 미팅 반영한 SR 확장 필요 |
| 아키텍처 설계서 | 모듈 구성, 데이터 흐름, 외부 API | 일부 문서 | OpenAI/NCP/로컬 저장 흐름 명확화 필요 |
| 상세 설계서 | 화면/모듈별 설계 | 산재 | step별, 결과 저장, AAC, 노래훈련 상세 설계 필요 |
| 단위 시험 | 함수/모듈 단위 테스트 | `runDeterministicChecks.ts` 일부 | 테스트 케이스 ID 체계 강화 |
| 통합 시험 | STT/안면/시선/AAC/DB/리포트 흐름 | 일부 수동 확인 | 자동/수동 테스트 절차서 필요 |
| 시스템 시험 | 사용자 시나리오 end-to-end | 부족 | 환자/치료사 역할별 시나리오 필요 |
| 위험 분석 | 위해요인, 통제, 잔여위험 | `risk-management-file.md` | 요구사항/검증 링크 보강 |
| 검증 보고서 | V&V 결과 요약 | `test:vnv`, export API | KTL 제출용 포맷 정리 필요 |

### 개발 작업

- 기존 `requirements.ts`를 기준으로 SR ID를 확장
- 각 SR에 다음 필드를 연결
  - 관련 위험 ID
  - 설계 문서 ID
  - 코드 위치
  - 테스트 케이스 ID
  - 검증 결과
- `traceability-matrix.md`를 KTL 미팅 기준으로 재작성
- `npm run test:vnv` 결과가 문서로 자동 export 되도록 유지

## 4. AI 검증 준비

KTL 답변 기준으로, OpenAI API를 쓰더라도 검증은 필요하다.

검증 대상은 “OpenAI 모델 자체”가 아니라 **우리 제품이 OpenAI API를 포함한 전체 파이프라인으로 만든 최종 출력**이다.

### 개발 작업

| 항목 | 필요한 구현/문서 | 상태 |
| --- | --- | --- |
| 테스트셋 분리 | 학습/튜닝/검증/시험 데이터 분리 규칙 | 문서 필요 |
| 정답지 정의 | 목표 발화, 허용 발음, SLP 판정 기준 | 부족 |
| 메트릭 계산 | WER/CER, 정확도, 민감도, 특이도, 분포 | 일부 가이드 있음 |
| 결과 export | CSV/JSON 결과표 | 일부 구현 필요 |
| 모델 버전 추적 | OpenAI 모델명, API 날짜, prompt, 분석 버전 | 일부 있음 |
| 실패 케이스 기록 | STT 실패, 네트워크 실패, 무음, 잡음 | 보강 필요 |

### 개발 원칙

- WASM-STT는 운영 경로에서 제외한다.
- 운영 STT는 서버 프록시 기반 OpenAI Whisper 경로로 명확히 문서화한다.
- 원본 음성 저장/전송 여부는 실제 구현 기준으로 문서화한다.
- 결과 리포트는 “AI 판단”이 아니라 “치료사 검토용 참고 결과”로 표시한다.

## 5. 사이버보안 개발 항목

KTL은 사이버보안을 마지막 점검이 아니라 설계 요구사항으로 넣어야 한다고 설명했다.

현재 프로젝트에는 35개 사이버보안 항목 정의가 있으나, KTL 체크리스트를 받으면 다시 매핑해야 한다.

### 개발 작업

- KTL 체크리스트 수령 후 `source-cybersecurity-requirements.md` 업데이트
- 각 보안 요구사항을 SR-* 또는 SEC-* 요구사항으로 등록
- 다음 항목은 실제 구현 증거가 필요
  - 로그인 시도 제한
  - 비밀번호 정책
  - 세션 만료 정책
  - 역할 기반 접근제어
  - 감사 로그
  - 전송 암호화
  - 저장 데이터 보호
  - SBOM/SOUP 목록
  - 취약점 대응 절차

  - 채점 로직 변경 있음/없음

  - 변경 내용
  - 영향 범위
  - 재시험 필요 여부
  - 변경심사 가능성
