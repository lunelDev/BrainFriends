# 계정 / 기관 / 매칭 재설계안

## 목적

이 문서는 BrainFriends를 `기관 - 치료사 - 사용자 - 담당 치료사 매칭` 구조로 재정리하기 위한 기준 문서입니다.

현재 서비스는 `회원가입`, `기관 등록`, `치료사 승인`, `담당 치료사 연결`이 일부 섞여 있습니다.  
운영형 서비스와 DTx / SaMD 대응 구조로 가려면 아래 기준으로 분리해야 합니다.

## 핵심 원칙

1. `기관`은 회원이 아니라 조직 데이터입니다.
2. `치료사`와 `사용자`는 사람 계정입니다.
3. 치료사는 반드시 기관 소속 승인 상태를 가져야 합니다.
4. 사용자는 가입 후 기관/치료사를 선택해 연결되거나, 기관 관리자 승인으로 연결됩니다.
5. 승인되지 않은 기관, 승인되지 않은 치료사, 승인되지 않은 소속 관계는 검색/노출되면 안 됩니다.

## 목표 서비스 구조

```text
기관(Institution)
 ├─ 기관 구성원(InstitutionMember)
 │   ├─ 기관 대표 / 관리자
 │   └─ 치료사
 └─ 사용자-치료사 매칭(UserTherapistMapping)
     ├─ 사용자(User)
     └─ 담당 치료사(Therapist)
```

## 회원 유형

회원가입 첫 화면에서는 아래 2개만 둡니다.

- 일반 회원가입
- 치료사 회원가입

설명:
- `기관 등록`은 회원가입 유형이 아니라 `치료사 온보딩` 또는 `기관 등록 요청` 단계에서 처리합니다.
- `기관` 자체는 조직 엔티티이므로 사람 계정과 분리합니다.

## 권장 플로우

### 1. 일반 회원가입

```text
일반 회원가입
→ 기본 정보 입력
→ 계정 생성
→ 기관 검색
→ 치료사 검색/선택
→ 담당 치료사 연결 요청
→ 치료사 또는 기관 관리자 승인
```

일반 회원가입 입력 항목:

- 이름
- 휴대폰 번호
- 이메일 또는 로그인 ID
- 비밀번호
- 생년월일(선택)
- 성별(선택)

중요:
- 현재처럼 일반 회원가입 폼 안에서 기관/치료사를 함께 고정하는 방식은 줄이는 것이 좋습니다.
- 더 안전한 구조는 `계정 생성`과 `담당 치료사 연결`을 분리하는 것입니다.

### 2. 치료사 회원가입

```text
치료사 회원가입
→ 기본 정보 입력
→ 자격 정보 입력
→ 기관 연결 방식 선택
   ├─ 기존 기관 선택
   └─ 새 기관 등록 요청
→ 기관 소속 승인 대기
→ 자격 승인 대기
→ 활성화
```

치료사 회원가입 입력 항목:

- 이름
- 휴대폰 번호
- 이메일
- 비밀번호
- 자격/면허 번호
- 자격증 파일
- 발급기관
- 발급일
- 직군
- 전문 분야
- 소개글(선택)
- 2차 인증 방식
- 역할(Role)
- 법적 동의

### 3. 1인 기관 플로우

1인 기관은 별도 체크 흐름이 필요합니다.

```text
치료사 회원가입
→ "내가 기관 대표이자 치료사입니다" 체크
→ 기관 정보 입력
→ 기관 생성 요청
→ 본인을 OWNER + THERAPIST로 연결
→ 기관 승인 + 치료사 승인 후 활성화
```

중요:
- 1인 기관은 `기관 생성`, `치료사 생성`, `기관 소속 생성`을 한 번에 처리해야 합니다.

## 기관 등록 화면 기준

기관 등록은 사람 계정 생성이 아니라 심사형 등록 프로세스로 다룹니다.

필수:

- 기관명
- 사업자등록번호
- 대표자명
- 기관 유형
- 주소 / 상세주소 / 우편번호
- 연락처
- 요양기관번호
- 사업자등록증 업로드
- 개설신고필증 업로드

권장:

- 홈페이지
- 기관 소개
- 진료과목
- 의료기관 코드

## 상태값 기준

### 기관 상태

- `PENDING`
- `APPROVED`
- `REJECTED`

### 치료사 자격 상태

- `PENDING`
- `APPROVED`
- `REJECTED`

### 기관 소속 상태

- `PENDING`
- `APPROVED`
- `REJECTED`

### 사용자-치료사 매칭 상태

- `PENDING`
- `APPROVED`
- `REJECTED`
- `ENDED`

## 사용자 검색 노출 조건

검색 결과에 치료사가 노출되려면 아래를 모두 만족해야 합니다.

1. 치료사 계정 상태가 `ACTIVE`
2. 치료사 자격 상태가 `APPROVED`
3. 공개 여부가 `true`
4. 기관 소속 상태가 `APPROVED`
5. 기관 상태가 `APPROVED`

즉, 아무나 가입했다고 바로 검색 노출되면 안 됩니다.

핵심 엔티티는 아래 5개입니다.

1. `users`

4. `institution_members`
5. `user_therapist_mappings`

## 현재 코드와의 차이

현재 코드 기준으로 어긋나는 부분:

### 1. 일반 회원가입이 가입 중에 기관/치료사를 바로 고정

현재는 `/signup`에서 일반 회원가입 시 기관과 담당 치료사를 선택합니다.  
목표 구조 기준으로는 `계정 생성`과 `담당 치료사 연결`을 분리하는 편이 더 안전합니다.

### 2. `institution_members` 개념이 코드 전반에 아직 고정되지 않음

현재는 `organization_id`, `approval_state`, `therapist_patient_assignments` 중심으로 흩어져 있습니다.  
기관 소속은 별도 엔티티로 분리하는 것이 맞습니다.

### 3. `user_therapist_mappings`가 요청/승인형으로 분리되지 않음

현재 일반 회원가입 시 선택한 치료사와 즉시 연결하는 흐름이 남아 있습니다.  
의료 서비스 관점에서는 `PENDING → APPROVED` 구조가 더 안전합니다.

### 4. 1인 기관 전용 플로우 없음

현재는 치료사 가입과 기관 등록이 분리되어 있으나,  
`기관 대표이자 치료사`를 위한 예외 흐름은 없습니다.

## 현재 코드 기준 권장 변경 순서

### 1차

1. `기관은 조직 엔티티`라는 기준을 문서/DB 기준으로 고정
2. `institution_members` 테이블 도입
3. `user_therapist_mappings` 도입
4. 승인 상태 enum을 서비스 전반에서 통일

### 2차

5. 일반 회원가입에서 기관/치료사 선택을 가입 후 단계로 이동
6. 치료사 회원가입에서 기관 연결을 `기존 기관 선택 / 새 기관 등록 요청`으로 분리
7. 1인 기관 체크 플로우 추가

### 3차

8. 검색 노출 조건을 승인 상태 기반으로 강제
9. 관리자 승인 UI를 `기관 / 치료사 / 소속 / 매칭` 4축으로 재정리
10. RBAC를 기관 소속/역할 기준으로 강화

## 한 줄 결론

BrainFriends는 `기관은 조직`, `치료사/사용자는 사람 계정`, `치료사는 기관 소속 승인`, `사용자-치료사 연결은 별도 매칭` 구조로 재설계하는 것이 맞습니다. 현재 코드도 이 기준으로 단계적으로 정리해야 운영형 서비스와 DTx / SaMD 요구사항에 맞습니다.
