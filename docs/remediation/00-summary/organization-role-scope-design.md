# 병원 / 역할 / 배정 구조 설계안

## 목적

이 문서는 BrainFriends를 실제 운영형 구조로 확장할 때 필요한 `병원(기관)`, `역할`, `치료사-사용자 배정` 기준을 정리한 설계안입니다.

현재 구조에서는 치료사 화면이 `전체 사용자 데이터`를 보는 방향으로 흘러갈 수 있으므로, 이 문서는 그 문제를 막기 위한 기준 문서입니다.

## 현재 문제

### 현재 구조의 위험

- 치료사 화면은 분리되어 있으나, 조회 데이터는 사실상 `admin 전체 조회` 구조를 재사용하고 있습니다.
- 이 상태에서는 치료사 계정이 담당하지 않은 사용자 정보까지 볼 위험이 있습니다.
- 실제 운영에서는 치료사별 담당 사용자 범위를 제한하지 않으면 안 됩니다.

### 결론

치료사 UI를 더 고도화하기 전에 먼저 아래 구조를 넣어야 합니다.

1. 병원(기관) 소속
2. 사용자/치료사 역할 구분
3. 치료사-사용자 배정 관계
4. 치료사 API 범위 제한

## 목표 구조

### 1. 병원(기관) 단위

- 하나의 사용자와 치료사는 특정 병원(기관)에 소속됩니다.
- 병원이 다르면 기본적으로 서로 데이터를 볼 수 없습니다.

### 2. 역할 단위

- `user`
- `therapist`
- `admin`

설명:
- `user`: 훈련 수행 및 개인 결과 확인
- `therapist`: 담당 사용자 결과 확인 및 후속 조치
- `admin`: 병원 또는 시스템 운영 관리

### 3. 치료사-사용자 배정 단위

- 치료사는 같은 병원 안에서도 `배정된 사용자`만 조회할 수 있어야 합니다.
- 전체 사용자 목록은 `admin`만 볼 수 있어야 합니다.

## 권장 DB 구조

### 1. 기관 테이블

예시:

- `organizations`
  - `organization_id`
  - `organization_name`
  - `organization_code`
  - `is_active`
  - `created_at`
  - `updated_at`

역할:
- 병원/기관 단위의 기준 테이블

### 2. 사용자 계정 테이블 확장

현재 `app_users`에 아래 컬럼 추가 권장:

- `organization_id`
- `approval_state`

설명:
- 모든 계정은 소속 기관을 가져야 함
- 치료사 계정은 승인 대기 상태를 둘 수 있음

### 3. 사용자 프로필 연결

`patient_pii`, `patient_pseudonym_map` 등과 `organization_id` 연결 필요

설명:
- 사용자는 특정 기관 소속이어야 함
- 치료사는 같은 기관 사용자만 배정 가능해야 함

### 4. 치료사-사용자 배정 테이블

예시:

- `therapist_patient_assignments`
  - `assignment_id`
  - `organization_id`
  - `therapist_user_id`
  - `patient_id`
  - `assigned_at`
  - `assigned_by`
  - `is_active`

설명:
- 치료사와 사용자의 실제 담당 관계를 저장
- 치료사 화면은 이 테이블 기준으로만 조회해야 함

## 회원가입/계정 생성 흐름

### 1. 사용자 가입

필요 항목:
- 병원 검색 또는 병원 코드 입력
- 기본 정보 입력
- 계정 생성

권장 흐름:
1. 병원 검색
2. 병원 선택
3. 사용자 기본정보 입력
4. 회원가입 완료

### 2. 치료사 가입

필요 항목:
- 역할 선택: `치료사`
- 병원 검색 또는 병원 코드 입력
- 자격 정보 또는 관리자 승인 대기

권장 흐름:
1. 역할 선택
2. 병원 선택
3. 치료사 계정 신청
4. 관리자 승인 후 활성화

### 3. 관리자 계정

관리자는 아래 작업 가능해야 함:
- 치료사 계정 승인
- 사용자-치료사 배정
- 배정 해제
- 기관 내 사용자 전체 조회

## 서버/API 변경 방향

### 1. 치료사 리포트 API

현재 문제:
- `therapist`도 `admin` 전체 조회 함수를 재사용하는 구조

변경 방향:
- `adminReportsDb.ts`와 별도로 `therapistReportsDb.ts` 분리
- 치료사 계정은 `therapist_patient_assignments` 기준 사용자만 조회

즉:
- `admin` → 전체 조회 가능
- `therapist` → 담당 사용자만 조회 가능

### 2. 치료사 상세 / 메모 / 결과 API

아래 API들도 스코프 제한 필요:
- `/api/therapist/reports`
- `/api/therapist/reports/note`
- 필요 시 `export` 계열

조건:
- 본인에게 배정된 사용자만 열람 가능
- 배정되지 않은 사용자면 `403` 또는 `not_assigned` 반환

### 3. 회원가입 API

`/api/auth/signup` 확장 필요:
- `organization_id`
- `requested_role`

또는
- 병원 검색 결과에서 선택한 조직 정보

### 4. 관리자 배정 API

추가 필요:
- 치료사에게 사용자 배정
- 배정 해제
- 현재 배정 목록 조회

## UI 변경 방향

### 1. 사용자 회원가입 화면

추가 필요:
- 병원 검색/선택
- 소속 기관 표시

### 2. 치료사 회원가입 또는 생성 화면

추가 필요:
- 역할 선택
- 병원 선택
- 승인 상태 표시

### 3. 관리자 화면

추가 필요:
- 사용자-치료사 배정 관리 화면

### 4. 치료사 화면

변경 필요:

- 검색도 담당 사용자 집합 안에서만 수행

## 현재 반영 상태

- 치료사 리포트 API와 치료사 메인 대시보드는 이제 관리자 전체 조회 함수를 직접 쓰지 않고 치료사 스코프 계층을 거치도록 분리했습니다.
- `therapist_patient_assignments` 테이블이 없는 환경에서는 치료사 계정에 `빈 목록`을 돌려주는 fail-safe 방식으로 동작합니다.
- 즉, 배정 구조가 실제로 도입되기 전에는 치료사가 전체 사용자 데이터를 보지 못하도록 막는 방향으로 먼저 고정했습니다.

## 한 줄 결론

지금처럼 치료사가 전체 사용자 데이터를 볼 수 있는 구조는 운영형으로 적절하지 않습니다. 먼저 `병원 소속 + 역할 구분 + 치료사-사용자 배정` 구조를 넣고, 그 기준으로 API와 회원가입 흐름을 다시 잡아야 합니다.
